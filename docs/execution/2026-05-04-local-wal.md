## Goal

Implement the approved local WAL design in staged checkpoints that keep the
durability primitive, serving integration, and cleanup/compaction path
independently reviewable.

The target end state is:

- a local `WalDomain` WAL SDK with a domain-scoped `WalProvider` /
  `ExportWal` boundary;
- durable append semantics where acknowledged writes have fsynced WAL records;
- ordered replay after `export_heads.checkpoint_wal_seq`;
- a `wal_durable` engine kind served through `WalDurableEngine`;
- an `ExportReadView` that overlays retained WAL records over committed
  snapshot state;
- lifecycle-triggered compaction, first triggered on clean close, that normally
  moves WAL records into committed COW tree state and then makes checkpointed
  WAL eligible for cleanup;
- recovery that remains correct when close-triggered compaction fails by
  replaying retained WAL records newer than the last published checkpoint.

## Why Split

This effort changes local filesystem durability, catalog engine/layout
semantics, request ownership, export serving state, restart recovery, and
lifecycle cleanup. A single implementation series would make it too easy to
mix the WAL primitive with serving behavior and compaction details.

The execution checkpoints are:

1. build and test the local WAL SDK/backend as a standalone primitive;
2. add `wal_durable` catalog/CLI/registry serving integration;
3. add lifecycle compaction and checkpoint-gated WAL cleanup, with clean close
   as the first trigger.

Only Series 1 is approved for immediate implementation. Series 2 must receive
a fresh next-series execution review before implementation. Series 3 is
expected to need focused compaction/cleanup design before `$plan-series`.

## Series 1: Local WAL SDK And Backend

Depends on: none

Design coverage:
`docs/plans/2026-05-04-local-wal.md`, Part 1.

Stable checkpoint: the repository has a tested local WAL primitive that can be
opened independently of NBD serving, append durable write records, report
high-watermark bounds, replay retained records through a cursor-shaped API,
repair a final torn tail, reject interior corruption, and perform explicit
prefix cleanup when the caller has already proven cleanup safety.

Review focus: WAL source-of-truth boundaries, `WalDomain` sequence assignment,
local path containment and namespace encoding, segment naming as backend
detail, file and directory sync ordering, Tokio-safe filesystem work, record
framing, CRC validation, tail repair policy, interior corruption handling,
bounds semantics, replay cursor shape, and keeping `prune_through` as a
backend cleanup primitive rather than a lifecycle policy.

Done means: `runtime.wal_dir` exists in config and generated defaults point to
`~/.cache/nbd/wal`; `WalProvider`, `ExportWal`, `WalRequest`,
`WalRecord`, `WalBounds`, `WalReplay`, `WalPruneResult`, and `WalDomain` exist
behind the server WAL module; `OpenWal` carries `WalDomain`;
`LocalWalProvider` opens one export-id-backed `LocalExportWal`; local append
assigns contiguous `WalSeq` values per `WalDomain` under an append mutex,
writes an inline payload frame, fsyncs according to the local durability
policy, and updates in-memory bounds only after durability succeeds; local
reopen scans segments, repairs only a final corrupt or partial tail, and
rejects sequence gaps or interior corruption; replay returns retained records
in strict sequence order through the cursor API; `prune_through` never deletes
records above the requested sequence and reports actual cleanup; tests cover
append, reopen, bounds, replay, torn tail, interior corruption, and pruning
behavior.

Stable checkpoint: `wal_durable` exports can be created explicitly, opened by

server restart by replaying retained WAL records newer than the catalog

Review focus: catalog schema truth, engine/layout naming, current-head

Done means: the catalog domain model, Prisma schema, SQLite catalog, CLI, and
tests accept `wal_durable` and `cow_immutable_tree`; creating a `wal_durable`
export initializes an empty COW-compatible head with `root_node_id = null` and
`checkpoint_wal_seq = 0`; generated config includes `runtime.wal_dir`;
`NbdServer::start` constructs a `WalProvider` service facade; `ExportFactory`

`WalDurableEngine` implements `ExportEngine`, owns the WAL handle and

`WalDurableAdmissionPolicy` with normal byte-range admission; the export API
supports owned admitted-request extraction so WAL writes can move payload bytes
without cloning or dropping admission early; reads combine WAL overlay with

## Series 3: Lifecycle Compaction And Cleanup

Depends on: Series 2

Design coverage:
`docs/plans/2026-05-04-local-wal.md`, Part 3. This series is expected to need
a focused compaction/cleanup design or a materially expanded execution review
before implementation.

Stable checkpoint: the repository has a `CompactionManager` boundary that can
compact a `wal_durable` export from durable catalog, WAL, tree, and blob
sources. The first trigger is clean close: it normally compacts retained WAL
records through the durable WAL high watermark into committed COW tree state,
publishes a new `export_heads` checkpoint, and makes checkpointed WAL segments
eligible for cleanup. Failed close-triggered compaction leaves the export
recoverable by WAL replay on the next open.

Review focus: compaction as an independent lifecycle event rather than an
engine method, at most one active compaction per export, quiescing writes
before close-triggered compaction target selection, durable WAL high-watermark
capture, active compaction semantics where writes after the captured high
watermark remain in WAL, WAL prefix checkpoint semantics, immutable COW tree
output, catalog checkpoint publication, read-view overlay retirement, failure
ordering, timeout behavior, active read-view safety before physical cleanup,
and restart recovery after both successful and failed close-triggered
compaction.

Done means: `CompactionManager` can run from durable sources without depending
on a live `WalDurableEngine`; it enforces at most one active compaction per
export; the close path for `wal_durable` stops accepting new writes, lets
admitted writes finish or fail, captures a durable WAL high watermark, asks
`CompactionManager` to compact records from the previous checkpoint through
that high watermark into committed COW tree state, publishes the new
root/checkpoint through `export_heads`, updates or retires read-view overlay
entries represented by the checkpoint, and calls local WAL cleanup only after
checkpoint coverage and active read-view safety are proven; tests prove restart
after successful cleanup and restart after failed compaction before checkpoint
publication.

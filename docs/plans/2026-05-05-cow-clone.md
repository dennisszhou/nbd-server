Title: COW Export Clone
Date: 2026-05-05
Status: approved

Problem
- The catalog and WAL/COW tree architecture now supports immutable committed
  roots, but there is no operator command to create a new export from an
  existing committed root.
- `nbdcli` and `nbd-server` both directly construct `SQLiteExportCatalog`.
  That makes `ExportCatalog` less useful as the provider boundary and would
  spread provider-specific knowledge into clone call sites.
- Clone semantics are sharp because `wal_durable` serving state is:

```text
published COW tree at export_heads.checkpoint_wal_seq
  + retained WAL records newer than that checkpoint
```

  A catalog-only clone can copy the published tree. It cannot see or include a
  running server's current in-memory read view or uncheckpointed WAL overlay.

Goal
- Add `nbdcli clone SOURCE DESTINATION` for COW exports.
- Make clone an `ExportCatalog` operation owned by the control plane.
- Keep provider selection inside `nbd-control-plane` so binaries depend on
  catalog traits and handles rather than `SQLiteExportCatalog`.
- Implement v1 clone as an O(1) committed-checkpoint clone:
  - source must be active `wal_durable`;
  - source head must use `cow_immutable_tree`;
  - source head must have a committed non-null root;
  - destination gets a new export identity and WAL domain;
  - destination points at the source's committed COW root;
  - no blob bytes, WAL records, or tree nodes are copied.
- Keep clone readable and writable through the existing `WalDurableEngine`
  open, replay, and compaction paths.

Constraints
- The active catalog model uses `exports` plus one current `export_heads` row.
  Do not reintroduce export generations.
- `root_node_id = null` is a valid all-zero committed tree, but v1 clone
  rejects it. A clone destination should mean "copied an existing snapshot",
  not "created another empty export".
- `checkpoint_wal_seq` is per export WAL state. A cloned export starts its own
  WAL at checkpoint zero even when its root came from another export.
  The copied root is the destination's imported baseline, not a destination
  WAL prefix.
- Current `WalDomain` is keyed by export id. A cloned export therefore gets an
  independent WAL automatically when it gets a new export id.
- `simple_mutable_tree` rows and mutable blob files are export-private and are
  not clone-safe.
- The current prototype has no durable active-export lease or server RPC for
  forcing compaction before clone.

Non-goals
- Live clone of uncheckpointed source WAL records.
- Source freezing, source fencing, active export leases, or auth/client owner
  integration.
- Manual compaction commands such as `nbdcli compact`.
- Clone support for `memory` or `simple_durable`.
- Physical blob copy, tree-node copy, clone lineage tracking, snapshot listing,
  or garbage collection.
- Schema redesign or renaming `owner_export_id` in this series.

End state
- `nbdcli clone source destination` creates a new active `wal_durable` export
  that serves the source's latest committed COW root.
- The CLI output states that clone copied the committed checkpoint and does
  not include source WAL records newer than that checkpoint.
- If the source has no committed COW root, `nbdcli clone` fails with a clear
  message that the source snapshot is empty.
- `nbdcli` and `nbd-server` open catalog services through a shared
  control-plane factory instead of importing `SQLiteExportCatalog` directly.
- The destination can be opened through the existing server path. Reads see the
  shared committed root plus the destination's own WAL overlay.
- Writes to the destination append only to the destination WAL and do not
  affect the source.
- Later destination compaction publishes a destination-owned COW root while
  retaining shared immutable blobs for unchanged chunks.

Proposed approach
- Introduce a catalog service opener in `nbd-control-plane`.

```rust
pub struct CatalogHandle {
    export_catalog: Arc<dyn ExportCatalog>,
    simple_tree_store: Arc<dyn SimpleTreeMetadataStore>,
    cow_tree_store: Arc<dyn CowTreeMetadataStore>,
}

pub async fn open_catalog(url: &CatalogUrl) -> Result<CatalogHandle>;
```

  The SQLite implementation remains concrete inside `nbd-control-plane`.
  Tests that need direct SQLite access for migrations may still use
  `SQLiteExportCatalog`, but production binaries should not perform provider
  selection themselves.

- Add clone request and result records to the catalog model.

```rust
struct CloneExport {
    source: ExportName,
    destination: ExportName,
}

struct CloneExportResult {
    source: ExportMeta,
    destination: ExportMeta,
}
```

  Returning the source metadata is intentional. It lets `nbdcli` print the
  checkpoint and root that were cloned without issuing a second, racy inspect.

- Add clone to the catalog trait.

```rust
trait ExportCatalog {
    async fn clone_export(&self, request: CloneExport)
        -> Result<CloneExportResult>;
}
```

- Implement SQLite clone as one transaction.

```text
begin
  load source exports + export_heads row by source name
  require source state = active
  require source engine_kind = wal_durable
  require source layout_kind = cow_immutable_tree
  require source root_node_id is not null
  generate destination export id
  insert destination exports row:
    name = request.destination
    engine_kind = wal_durable
    block_size = source.block_size
    state = active
  insert destination export_heads row:
    export_id = destination id
    layout_kind = cow_immutable_tree
    root_node_id = source.root_node_id
    size_bytes = source.size_bytes
    checkpoint_wal_seq = 0
  return source metadata and destination metadata
commit
```

  A source `root_node_id = null` is rejected as an empty committed snapshot. If
  the source has uncheckpointed WAL records, those records are intentionally
  not part of the clone; close-time or later compaction must publish a
  committed root before clone can copy that state.

- Adjust COW tree metadata validation for sharing.

  `simple_mutable_tree` validation must continue to require
  `owner_export_id == export_id` because simple blobs are mutable and
  export-private.

  `cow_immutable_tree` validation must treat `owner_export_id` as provenance
  for who created the immutable node, not as exclusive ownership. A cloned
  export may point at a root and leaf nodes originally created by the source
  export. Validation should still require:
  - layout is `cow_immutable_tree`;
  - root node kind and level are valid;
  - leaf node kind, level, storage kind, and length are valid;
  - chunk indexes fit inside the destination export size.

- Add `nbdcli clone`.

```text
nbdcli clone SOURCE DESTINATION
```

  Human output should include the destination name, source name, source
  checkpoint, and a short committed-checkpoint note. `clone` does not add
  `--json` in v1. Scripts that need structured state can call
  `nbdcli inspect DESTINATION --json` after clone, matching the current
  imperative `create` and `delete` command style.

Data model / API shape
- Authoritative state:
  - `exports` owns source and destination identities, names, engine kinds,
    block sizes, lifecycle states, and timestamps.
  - `export_heads` owns each export's serving root, size, layout, and per-export
    checkpoint.
  - `tree_nodes`, `tree_edges`, and `tree_leaf_refs` own immutable COW metadata
    reachable from committed roots.
  - WAL files remain per export id through `WalDomain::for_export_id`.

- Derived or cached state:
  - `ExportReadView` is reconstructed from the destination COW head plus the
    destination WAL.
  - CLI output is derived from `CloneExportResult`.

- Request structures:

```rust
impl CloneExport {
    pub fn new(source: ExportName, destination: ExportName) -> Result<Self>;
    pub fn source(&self) -> &ExportName;
    pub fn destination(&self) -> &ExportName;
}

impl CloneExportResult {
    pub fn source(&self) -> &ExportMeta;
    pub fn destination(&self) -> &ExportMeta;
}
```

- Catalog errors:
  - missing source returns `ExportNotFound`;
  - deleted source returns `ExportDeleted`;
  - existing destination returns `ExportAlreadyExists`;
  - source `root_node_id = null` returns `InvalidField` with a clear reason
    that the source committed snapshot is empty;
  - unsupported source engine or layout returns `InvalidField` with a
    clone-specific reason.

Invariants
- Clone is a catalog transaction.
- Clone never mutates the source export.
- Clone never copies source WAL records.
- Clone never copies blob bytes.
- Clone never creates tree metadata for already committed source chunks.
- Destination `exports.id` is new.
- Destination `exports.name` must not already exist, including deleted exports
  while the catalog has a unique `exports.name` constraint.
- Source `root_node_id` must be non-null.
- Destination `engine_kind = wal_durable`.
- Destination `layout_kind = cow_immutable_tree`.
- Destination `root_node_id = source.root_node_id`, which is non-null.
- Destination `size_bytes = source.size_bytes`.
- Destination `block_size = source.block_size`.
- Destination `checkpoint_wal_seq = 0`.
- Destination WAL replay starts from the destination WAL, not the source WAL.
- COW metadata can be shared across exports by root reachability.
- Simple mutable metadata remains export-private.

Lifecycle / operational contract
- Cloning an active source is allowed, but it clones only the latest committed
  source head visible in the catalog transaction.
- Close-time compaction remains the current way to make recent writes visible
  to a subsequent catalog-only clone.
- If close-time compaction fails, acknowledged source writes remain durable in
  source WAL but are not included in a clone until compaction later publishes a
  covering root.
- Opening the clone uses the normal `LocalExportRegistry` path:

```text
open destination name
  -> load destination ExportDescriptor
  -> open WalDomain::for_export_id(destination_id)
  -> load destination CowTreeSnapshot
  -> initialize ExportReadView from shared committed root
  -> replay destination WAL records after checkpoint 0
```

Alternatives considered
- Clone live WAL overlay.
  - Rejected for v1. It requires server coordination or WAL-service semantics
    that the current `nbdcli` catalog path does not have.
- Require source to be closed.
  - Rejected for v1 because there is no durable active-export lease yet. A
    process-local registry check would not be a real correctness boundary.
- Force compaction before clone.
  - Rejected for v1 because compaction is currently a server-side background
    lifecycle feature, not a control-plane command.
- Support `simple_durable` clone by copying mutable blobs.
  - Rejected. It would be a different copy operation, not O(1) COW clone, and
    would need direct-commit consistency rules.
- Return only destination `ExportMeta`.
  - Rejected. The source checkpoint is part of the operator-visible clone
    contract, and returning it from the same transaction avoids a racy follow-up
    inspect.

Migration / rollout
- No schema migration is required.
- `owner_export_id` keeps its current column name. For COW rows it is treated
  as provenance, not exclusive ownership. A future schema cleanup may rename or
  split that concept if GC needs richer reachability state.
- Architecture docs have been updated to describe clone as a current planned
  COW operation and to require a non-null source root.

Validation strategy
- Catalog model tests:
  - `CloneExport` validates non-empty source and destination names;
  - cloning an export to the same name is rejected by the request model.

- SQLite catalog tests:
  - clone of active `wal_durable` COW export succeeds;
  - destination receives copied size, block size, engine kind, layout, root,
    and checkpoint zero;
  - source metadata is unchanged;
  - destination duplicate is rejected;
  - source with `root_node_id = null` is rejected as an empty committed
    snapshot;
  - missing or deleted source is rejected;
  - `memory` and `simple_durable` sources are rejected;
  - cloned COW root can be loaded through `load_cow_tree` even when its nodes
    were created by the source export.

- CLI tests:
  - `nbdcli clone source destination` succeeds for a committed COW source;
  - `inspect destination --json` shows `wal_durable`,
    `cow_immutable_tree`, copied root, copied size, and checkpoint zero;
  - unsupported source errors are surfaced clearly.

- Server/protocol integration:
  - create `wal_durable` source;
  - write data through the userspace NBD client;
  - disconnect and wait for compaction to publish a non-null root;
  - clone source to destination through the catalog API or CLI;
  - open destination through the server and verify cloned data;
  - write to destination, flush, and verify source still reads original data;
  - disconnect destination and wait for destination compaction to publish from
    the shared source-owned base root;
  - reopen both exports and verify destination changes persisted while the
    source remains unchanged.

- Regression around sharing:
  - COW tree loading must accept shared immutable roots;
  - simple tree loading must still reject roots or leaves owned by another
    export.

Risks
- Operator confusion is the main risk. `clone` sounds like live snapshot, but
  v1 is committed-checkpoint clone. CLI output and docs must state this
  plainly.
- Rejecting `root_node_id = null` avoids creating a destination that looks like
  a successful clone but only contains an empty committed snapshot. A source
  with uncheckpointed WAL records still needs compaction before clone can
  include those writes.
- Relaxing COW owner validation too broadly could hide malformed metadata.
  The change must be limited to ownership/provenance and keep layout, kind,
  storage kind, length, and range checks.
- Destination compaction after clone exercises the trickiest lifecycle: a
  destination export starts with a source-owned root, then publishes its own
  root from that base. The integration test must cover that close/reopen path.
- Future GC must use reachability from all export heads, not
  `owner_export_id`, before deleting immutable COW nodes or blobs.
- If future resize lands before clone is revisited, clone must copy a coherent
  source head size from the transaction.

Open questions
- None.

Follow-up cleanup
- The CLI has mixed output styles: `list` and `inspect` support `--json`, while
  imperative commands such as `create`, `delete`, and v1 `clone` use human
  output. A later CLI polish pass should make structured output policy uniform
  across commands.

Design exit criteria
- The design is ready for `$review-plan` when we agree that v1 clone is
  committed-checkpoint only and that catalog provider selection belongs behind
  a control-plane factory.
- The design is not ready if v1 must include live source WAL records or force
  source compaction before clone.

Recommended next step
- If this design shape is acceptable, run `$plan-series` for one reviewable
  clone series.

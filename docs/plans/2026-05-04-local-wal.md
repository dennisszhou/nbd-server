Title: Local WAL
Date: 2026-05-04
Status: approved

# Problem

The server now has an opt-in direct-commit `SimpleDurableEngine`, but it still
does not have the WAL-backed durability boundary described by the long-term
architecture. In the mature service, replication or a WAL service would own
durability. For this repository, the next step is a deliberately dumb local
WAL that is correct enough to use as the primitive behind NBD writes.

The WAL must not turn into a broad storage engine. Its first job is narrow:
append write records for one export WAL domain, report a durable sequence only
after the record is forced to local storage, replay records one at a time into
`ExportReadView`, and allow safe prefix pruning after a durable checkpoint says
older records are represented elsewhere.

# Goal

Add a local per-export WAL design that:

- exposes a small export-scoped SDK that can later be backed by a remote WAL
  service;
- keeps the WAL provider facade narrow enough that a future provider can key on
  `(owner, export_name)` without changing `WalDurableEngine`;
- assigns monotonically increasing per-export `WalSeq` values in v1;
- returns from `append` only after the record is durable under the local sync
  policy;
- lets recovery replay records one at a time after a catalog checkpoint;
- supports background WAL cleanup after a committed checkpoint and active
  read-view safety prove that the prefix is no longer needed;
- hides local segment files, record framing, checksums, and tail repair behind
  the WAL boundary;
- keeps connection, runtime, admission, catalog, read-view, and storage
  responsibilities separated.

# Research Notes

PostgreSQL describes WAL's central rule as logging changes and flushing those
records to permanent storage before relying on changed data files. It also
uses monotonically increasing log positions and numbered segment files.

SQLite's WAL documentation reinforces the same append/checkpoint split:
writes append to the WAL, reads combine base state with WAL state, and
checkpointing transfers WAL contents into the base database. SQLite also shows
why WAL cleanup must respect active readers.

RocksDB's WAL docs match the boundary needed here: updates go to both memory
state and WAL, crash recovery reconstructs memory state from WAL, WAL files are
read in sequence order, and old WALs become deletable only after their data has
been flushed elsewhere. RocksDB's recovery modes are useful mainly for the
tail policy: tolerate a final incomplete or corrupt tail, but treat interior
corruption as real data loss.

The ARIES paper is broader than this implementation needs. The useful lesson
is not to import transaction undo or fine-grained rollback; it is to keep log
sequence, redo/replay, and checkpoint semantics explicit.

# Constraints

- The first WAL is local and keyed on `export_id`.
- The default generated WAL path should be under `~/.cache/nbd/wal`.
- The WAL is one logical log per export in v1, even if the local backend stores
  that logical log as multiple segment files for pruning.
- The WAL does not use `StorageEngine`; it has its own replaceable backend.
- WAL sequence numbers are per export in v1, not global across exports.
- `export_id` is the v1 local WAL namespace key. The facade should keep this
  replaceable so a future WAL service can key by `(owner, export_name)` when
  auth/client identity exists.
- `ExportAdmissionCtl` tickets remain volatile and must not be used for WAL
  replay or compaction.
- `ExportReadView` is serving state. It is required for correctness while an
  export is active, but it is reconstructable from catalog checkpoint state
  plus durable WAL.
- `prune_through` must not be called merely because records were applied to
  `ExportReadView`. Physical cleanup is safe only after committed state
  represents those records and the retention window gives active read views
  enough time to refresh to the latest checkpoint.
- Published COW roots are complete snapshots through their checkpoint. A clone
  of such a root starts with a new export WAL; it must not depend on the
  source export's current WAL.
- The current WAL for an export represents writes newer than that export's
  latest committed checkpoint. Close-triggered compaction should normally move
  that prefix into committed tree state, but correctness must not depend on
  that cleanup succeeding.
- The local backend must use async file APIs or blocking offload where needed
  so sync-heavy filesystem operations do not block Tokio core workers.
- The first version may load replay records into memory internally, but the
  public replay API must remain cursor-shaped.
- Zero-length writes are no-ops at the WAL-backed engine boundary. They should
  not allocate WAL sequence numbers.

# Non-goals

- Cross-machine durability or replication.
- A remote WAL service implementation.
- Group commit, batching, manual WAL flush, or async commit.
- Transaction rollback, undo records, savepoints, or ARIES-style transaction
  recovery.
- Flush marker records.
- Compression or external payload storage.
- Random `get_record(WalSeq)` lookup or a persistent record index.
- An automated background or external compaction scheduler. The boundaries
  should allow compaction to run outside close, but the first implementation
  triggers it at clean close for testability and demo clarity.
- S3-backed blob storage.
- Writer fencing or multi-host serving.
- Strong all-or-nothing semantics for protocol writes whose reply was lost
  during connection or process failure.

# End State

The first WAL-backed export path can open an export, construct a local
export-scoped WAL handle, replay durable records after
`export_heads.checkpoint_wal_seq`, apply them to `ExportReadView`, and serve
NBD reads and writes through the existing runtime/admission path.

On clean close, the WAL-backed export should trigger lifecycle compaction
through the durable WAL high watermark, publish a new `export_heads`
checkpoint, and then make now-checkpointed WAL segments eligible for
background cleanup. If close-triggered compaction fails or times out, close can
still complete: the next open must recover by loading the last committed
checkpoint and replaying the retained WAL records newer than that checkpoint.

The generated config includes:

```text
runtime.wal_dir = "~/.cache/nbd/wal"
```

An export with id `export-a` stores its local logical WAL under an encoded
local path component:

```text
~/.cache/nbd/wal/6578706f72742d61/
  0000000000000001.wal
  0000000000002049.wal
```

Segment names are local backend detail. The export-facing API sees one ordered
record stream.

The work should be built in three parts:

1. build and test the WAL SDK plus local backend as a standalone primitive;
2. consume that primitive from `WalDurableEngine` and wire it into serving;
3. add lifecycle-triggered compaction and background WAL cleanup, with clean
   close as the first trigger.

# Proposed Approach

Introduce a WAL SDK with two layers:

- `WalProvider`, which opens the WAL handle for one `WalDomain`;
- `ExportWal`, which is the export-scoped append/replay/prune boundary.

This uses the concrete `WalProvider` / `ExportWal` boundary from
`docs/architecture/wal-architecture.md`: callers depend on the WAL contract,
not on the local backend.

`WalProvider` should be treated as a standalone WAL service facade. The local
implementation can hand out an independent `ExportWal` instance per
`WalDomain`. In v1, that domain is backed by `export_id`. Long term, the same
boundary can be backed by a shared service client whose internal operation
looks more like `WalService.append(WalRecord)` and whose domain key is
`(owner, export_name)`.

The engine should depend on `ExportWal`, not on local files or service
placement. A future `RemoteWalProvider` can return an `ExportWal` backed by a
service without changing the write/read/flush path.

## Part 1: WAL SDK And Local Backend

Part 1 is only the WAL primitive. It should not introduce `wal_durable`,
`WalDurableEngine`, `ExportReadView`, registry integration, or protocol-visible
behavior.

Part 1 owns:

- `WalProvider`, `ExportWal`, and WAL domain structs;
- `LocalWalProvider` and `LocalExportWal`;
- `runtime.wal_dir` config loading and default generation;
- record encoding, decoding, and CRC-32C validation;
- local segment creation, append, sync, replay, tail repair, and pruning;
- focused WAL tests that prove append, replay, recovery, and prune behavior.

Part 1's stable checkpoint is a local WAL module that can be opened in tests,
append durable records, report bounds, replay records through the cursor API,
repair a final torn tail, reject interior corruption, and perform explicit
prefix cleanup when its caller has already proven cleanup safety. No NBD
export should depend on it yet.

## Part 2: WalDurableEngine Integration

Each active export should open one export-scoped WAL handle while constructing
the runtime. In code, the active export boundary is the shared
`ExportRuntimeHandle` plus its `ExportEngine`, not the TCP connection. That
means:

- `NbdServer::start` constructs a `WalProvider` service facade from
  `runtime.wal_dir`;
- the active export construction path receives that service facade and opens
  the WAL for `wal_durable` exports;
- replay runs before the runtime becomes `Active`;
- `WalDurableEngine` owns the opened `ExportWalHandle` and builds
  `ExportReadView`;
- same-owner connections reuse the existing active runtime, engine, WAL handle,
  and read view.

The connection and runtime layers should not open WAL files. They should keep
submitting `ExportRequest` jobs through the existing `ExportRuntime` API.

The engine should be named `WalDurableEngine` to match the current concrete
engine naming convention. Architecturally, `wal_durable` is a new export type;
in code today that type is expressed by `ExportEngineKind::WalDurable` plus a
concrete `ExportEngine` implementation.

The construction shape should keep backend creation behind a concrete
`ExportFactory` so the registry does not become the permanent owner of every
backend dependency. This does not need a trait in v1:

```rust
pub struct LocalExportRegistry {
    catalog: Arc<dyn ExportCatalog>,
    factory: Arc<ExportFactory>,
    active: Mutex<HashMap<ExportName, ActiveExportState>>,
}

pub struct ExportFactory {
    config: ServerConfig,
    blob_dir: PathBuf,
    catalog: Arc<dyn ExportCatalog>,
    simple_tree_store: Arc<dyn SimpleTreeMetadataStore>,
    cow_tree_store: Arc<dyn CowTreeMetadataStore>,
    wal_provider: Arc<dyn WalProvider>,
}

impl ExportFactory {
    pub async fn open_export(
        &self,
        descriptor: ExportDescriptor,
    ) -> Result<OpenedExport>;
}

pub struct OpenedExport {
    pub runtime: ExportRuntimeHandle,
    pub export_id: ExportId,
    pub engine_kind: ExportEngineKind,
}
```

`ExportFactory` owns concrete construction dependencies such as the blob
directory, durable tree stores, runtime config, catalog head access, and
`WalProvider`. It is still the local process factory; the name stays short
because there is no second factory implementation today. A future remote WAL
implementation should replace the provider held by the factory, not the
registry lifecycle rules. If another factory implementation actually appears
later, a trait or rename can be extracted then.

`ActiveExport` does not need a separate WAL field because the engine owns the
WAL handle:

```rust
struct ActiveExport {
    owner: ExportOwner,
    runtime: ExportRuntimeHandle,
    connections: usize,
}
```

The open path should split stable export identity from the current serving
head. `ExportDescriptor` comes from the `exports` row only. Durable engines
load the latest engine-specific head/tree snapshot by `export_id` while they
open. That keeps background compaction from making an already loaded
`ExportMeta` head stale before the engine constructs its read view.

The WAL-backed export should be constructed from `ExportDescriptor`, the
already opened WAL handle, and a freshly loaded COW tree snapshot. It is still
the concrete `ExportEngine` implementation used by `ExportRuntime`, but it
owns its read-side state internally:

```rust
pub struct WalDurableEngine {
    name: ExportName,
    size_bytes: u64,
    block_size: u64,
    wal: ExportWalHandle,
    read_view: ExportReadView,
    write_lock: Mutex<()>,
}

impl WalDurableEngine {
    async fn open_with_cow_tree(
        descriptor: ExportDescriptor,
        wal: ExportWalHandle,
        catalog: Arc<dyn CowTreeMetadataStore>,
    ) -> Result<Self>;
}
```

`WalDurableEngine::open_with_cow_tree` initializes `ExportReadView` from the
latest COW tree snapshot, replays `wal.replay_after(checkpoint_wal_seq)`,
applies each `WalRecord`, and returns an engine only after recovery succeeds.

For v1, `WalDurableEngine::open` builds the read view with a zero-fill backing
reader because the WAL-backed head starts at an empty committed root. When
committed COW tree reading lands, `WalDurableEngine` can construct or receive a
read-only backing reader internally, but the registry should still not build
`ExportReadView` for it.

There is intentionally no blob writer, tree writer, or catalog writer in
`WalDurableEngine`. A read-only blob/tree reader may exist behind
`ExportReadView` once committed tree reads exist. For normal WAL-backed writes,
the export appends to WAL and then updates `ExportReadView`; it does not write
blobs or tree metadata.

The `write_lock` is intentionally conservative. It serializes
append-and-apply for WAL-backed writes so `ExportReadView` observes the same
sequence order assigned by the WAL. Admission still protects read/write
semantics, and a future ordered apply queue can relax this without changing the
engine API.

The open flow should be:

```text
LocalExportRegistry.open(name, owner)
  -> mark export Opening
  -> load ExportDescriptor from exports
  -> factory.open_export(descriptor)
       wal_durable path:
         derive WalDomain::for_export_id(descriptor.id())
         wal_provider.open_export(OpenWal { domain })
         load latest CowTreeSnapshot from export_heads/tree metadata
         WalDurableEngine::open_with_cow_tree(descriptor, wal, snapshot)
         wrap engine in SerialExportRuntime or ConcurrentExportRuntime
  -> mark export Active
```

The `Opening` state must remain in the registry while WAL replay runs. That
prevents another connection from opening the same export and assigning
sequence numbers against the same local WAL.

Part 2's stable checkpoint is a `wal_durable` export kind that uses the proven
WAL handle for request-path writes, constructs `ExportReadView` internally,
serves reads from WAL overlay plus committed backing state, and proves restart
recovery through the NBD protocol tests.

## Part 3: Lifecycle Compaction And WAL Cleanup

Part 3 adds the cleanup path that keeps the local WAL from growing forever.
Compaction is a lifecycle event, not an engine method. It can run at clean
close, on an operator request, or from a future background scheduler. The first
implementation should trigger compaction at clean close because that gives a
small, deterministic validation path.

Close-triggered compaction should enqueue background maintenance work. It is
best-effort for close correctness but correctness-critical for cleanup:

```text
connection close or NBD_CMD_DISC
  -> stop accepting new writes for the active export
  -> wait for admitted writes to finish or fail
  -> capture durable WAL high watermark H
  -> enqueue CompactionJob(export_id, through = H, reason = CleanClose)
  -> finish close
```

The close path does not wait for compaction to finish. Once admitted writes
have drained, the WAL is the durable recovery source and the `Export`,
`WalDurableEngine`, and `ExportReadView` may be dropped. Close captures `H`
only as a compaction target hint. The background worker rereads the current
database head and durable WAL bounds when it actually runs.

Background compaction runs from durable state only:

```text
CompactionManager worker receives CompactionJob(export_id, requested_H)
  -> ExportCatalog.load_export_by_id(export_id)
  -> read the current export_heads row
  -> open ExportWal for WalDomain::for_export_id(export_id)
  -> wal.bounds()
  -> target_H = min(requested_H, wal.bounds().last_durable)
  -> if target_H <= export_heads.checkpoint_wal_seq: no-op
  -> plan from the current catalog root/checkpoint/size/layout
  -> compact records checkpoint+1..target_H into committed COW tree state
  -> publish root/checkpoint through one catalog transaction
  -> make checkpointed WAL segments eligible for background cleanup
```

If compaction runs while an export remains active, it should capture
`H = wal.bounds().last_durable` at the start of that compaction attempt and
compact only `checkpoint+1..H`. Writes that become durable after `H` remain in
the WAL overlay and are replayed after the newly published checkpoint. The v1
close trigger can quiesce writes before enqueue only to make the captured
target deterministic; the worker still treats that target as a hint and
revalidates against the database and WAL.

If compaction fails or times out after acknowledged writes are already WAL
durable, the export may still close. The next open recovers by loading the
last published `export_heads` checkpoint and replaying retained WAL records
newer than that checkpoint.

Published COW roots are snapshot state complete through their checkpoint. A
future clone copies the committed root/checkpoint into a new export and starts
that destination export with a new WAL. The source export's current WAL is not
part of the clone snapshot.

Background cleanup may physically remove WAL segments only after the published
checkpoint covers those records and the retention policy says the segment is
old enough to prune.

The planned cleanup policy is time based. Active exports will periodically
refresh the latest catalog head and install newer checkpoints into
`ExportReadView`. The refresh interval must be shorter than the WAL retention
window. A serving export that cannot refresh before its current read view would
become older than the retention window must stop serving and force a reopen
from the current catalog head plus retained WAL.

That refresher and full retention cleanup loop are not required in Series 3.
Series 3 should keep the boundaries compatible with that solution. It may skip
physical pruning unless the test owns the serving process lifecycle tightly
enough to prove pruning is safe.

The same `CompactionManager` boundary should be usable after close and outside
the request path. It should depend on durable sources: the catalog head, the
export WAL service, and committed tree/blob writers. It should not depend on a
live `WalDurableEngine` instance.

Compaction must be idempotent. Repeated jobs and racing jobs are valid. A job
may write immutable blobs and tree metadata that later lose the publication
race; those unpublished objects are garbage for future GC. The catalog head
must never move backward and must never expose a partially published tree.

Catalog publication for compaction should be a single transaction that
compares the head used by the compaction plan with the current database head:

```text
publish_compaction(export_id, expected_base, new_root, compacted_through, batch)
  -> begin transaction
  -> load current export row and export_heads row
  -> if current.checkpoint_wal_seq >= compacted_through:
       return AlreadyCovered
  -> if current root/checkpoint/size/layout != expected_base:
       return StalePlan
  -> insert immutable COW tree metadata batch
  -> update export_heads:
       root_node_id = new_root
       checkpoint_wal_seq = compacted_through
       size_bytes = current.size_bytes
  -> commit
```

`AlreadyCovered` is a successful no-op. `StalePlan` is retryable by discarding
the unpublished compaction output and replanning from the current database
head through the original requested target.

Compaction work must not run on the export request queue. Tokio may remain the
process executor, but it is not the workqueue policy. `CompactionManager`
should own a separate bounded compaction queue:

```text
clean close
  -> try_enqueue CompactionJob into CompactionQueue
  -> release Export

CompactionQueue worker
  -> CompactionManager.compact_export(job)
```

Enqueue should have explicit outcomes:

```text
Queued
DroppedFull
ShuttingDown
```

Only `Queued` starts background work. `DroppedFull` and `ShuttingDown` are
not close failures because WAL remains the durable recovery source.

For v1, use a small fixed compaction worker count, likely one worker, and a
bounded pending queue. One worker limits background catalog, WAL replay, and
blob construction pressure without making correctness depend on serialization.
Duplicate or racing compactions must still be safe because the catalog
publication API is idempotent.

The queue API should hide the executor choice. The first implementation can
run that one worker as a Tokio task on the server's existing runtime. If
compaction starts competing with foreground request latency, the same
`CompactionQueue` boundary can move behind a dedicated current-thread Tokio
runtime on one OS thread, or a dedicated multi-thread runtime, without changing
the close path or compaction callers.

If enqueue fails because the compaction queue is full or shutting down, close
still succeeds after acknowledged writes are WAL-durable. The server should log
the skipped compaction, leave WAL records retained, and rely on a later clean
close, manual trigger, or future scheduler to retry cleanup.

Manager shutdown should be explicit: stop new enqueues, signal the worker, let
the currently running compaction job finish, drop any jobs still waiting in the
bounded queue, and join the worker. This preserves any in-progress catalog
publication while treating unstarted cleanup as retryable background work.

The compaction worker may call catalog, WAL, committed tree, and blob-store
APIs. Those APIs own their own blocking or storage I/O boundaries. Compaction
must not borrow request queue slots and must not run storage-heavy work by
spawning unbounded ad hoc Tokio tasks.

Cleanup safety should also be a narrow boundary. The long-term policy is
retention based: active exports refresh their read view to the latest catalog
checkpoint before the retention window expires, and background cleanup prunes
only checkpointed segments older than that window:

```text
CompactionManager
  -> WalRetentionPolicy.prune_cutoff(now)
  -> prune segments where:
       segment.max_seq <= published_checkpoint_wal_seq
       segment.closed_at <= prune_cutoff
```

The refresh worker and the full retention cleanup loop are future work, not
Series 3 requirements. Series 3 should not introduce a conflicting local-active
cleanup policy. If the series cannot prove retention safety in a test-owned
lifecycle, it should publish the checkpoint and leave WAL retained.

## Request Path

Write:

```text
ExportRuntime
  -> acquire write admission permit
  -> WalDurableEngine.write
  -> move the admitted ExportRequest payload into WalRequest
  -> validate range and construct WalRequest
  -> ExportWal.append(WalRequest)
  -> ExportReadView.apply_wal_record(WalRecord)
  -> reply success
```

This write path does not touch blob storage, committed tree metadata, or
`StorageEngine`. The durable record in the WAL is the source of truth until a
later compaction path publishes committed tree state that represents it.

The WAL path should not clone write payloads merely because
`AdmittedExportRequest` currently exposes only a borrowed request view. Add an
owned admitted-request extraction boundary that keeps the admission permit and
release logging alive while allowing `WalDurableEngine` to move the
`ExportRequest::Write` data into `WalRequest`.

Read:

```text
ExportRuntime
  -> acquire read admission permit
  -> ExportReadView.read(range)
  -> reply bytes
```

Reads first consult `ExportReadView`'s WAL overlay. Only bytes not covered by
WAL overlay records are read from the committed backing reader. In the first
WAL-backed engine, that backing reader can be all-zero because the catalog head
starts at an empty committed root.

Flush:

```text
ExportRuntime
  -> acquire flush admission permit
  -> WalDurableEngine.flush
  -> reply success
```

The current admission implementation makes flush a conservative export-wide
barrier. For the first WAL engine, that means a flush admitted into the engine
already follows earlier writes that finished WAL append and read-view apply.
No explicit WAL flush marker is needed.

## Startup Replay

Open:

```text
LocalExportRegistry.open
  -> load ExportDescriptor from ExportCatalog
  -> open ExportWal for WalDomain::for_export_id(export_id)
  -> load current COW tree snapshot from export_heads/tree metadata
  -> initialize ExportReadView from snapshot root/checkpoint
  -> replay records with seq > checkpoint_wal_seq
  -> apply each replayed record to ExportReadView in order
  -> construct WalDurableEngine and ExportRuntime
```

Replay is one record at a time at the API boundary:

```rust
let mut replay = wal
    .replay_after(checkpoint_wal_seq)
    .await?;

while let Some(record) = replay.next_record().await? {
    read_view.apply_wal_record(record)?;
}
```

The first local implementation may scan retained segments and collect replay
records into a `Vec` before returning `WalReplay`. That is intentionally dumb
but keeps the external recovery loop cursor-shaped. A later implementation can
make `WalReplay` stream from disk or page records from a service.

## Background WAL Cleanup

`prune_through(seq)` is a local backend cleanup primitive. It is not a
checkpoint operation, it does not change catalog state, and it is not called by
the write path.

Callers may invoke it only after durable committed state represents every WAL
record through `seq`, usually after compaction publishes:

```text
export_heads.checkpoint_wal_seq >= seq
```

The caller must also follow the retention policy. A serving view opened at an
older checkpoint can continue to serve correctly only while the WAL prefix it
needs is still retained. The planned policy is time-based retention: active
exports refresh to the latest checkpoint before the retention window expires,
and cleanup prunes only checkpointed segments older than that window. Until
that refresher and retention loop exist, compaction may publish checkpoints
without physically pruning WAL.

The local backend may retain records at or below `seq` when a segment straddles
the prune boundary. It must never delete a record with sequence greater than
`seq`.

# Data Model / API Shape

Use domain-scoped handles so normal callers do not pass the WAL domain on every
request.

```rust
pub type ExportWalHandle = Arc<dyn ExportWal>;

pub struct WalDomain {
    // v1 stores the domain as export_id. Future providers may map this
    // abstraction to (owner, export_name) without changing ExportWal callers.
    pub export_id: ExportId,
}

impl WalDomain {
    pub fn for_export_id(export_id: ExportId) -> Self;
}

pub struct OpenWal {
    pub domain: WalDomain,
}

#[async_trait::async_trait]
pub trait WalProvider: Send + Sync {
    async fn open_export(&self, request: OpenWal) -> Result<ExportWalHandle>;
}

#[async_trait::async_trait]
pub trait ExportWal: Send + Sync {
    async fn append(&self, request: WalRequest) -> Result<WalRecord>;

    async fn bounds(&self) -> Result<WalBounds>;

    async fn replay_after(&self, after: WalSeq) -> Result<WalReplay>;

    async fn replay_range(
        &self,
        after: WalSeq,
        through: WalSeq,
    ) -> Result<WalReplay>;

    async fn prune_through(&self, seq: WalSeq) -> Result<WalPruneResult>;
}
```

`WalDomain` is intentionally a small facade rather than a direct path string.
The first local backend uses `export_id`. A later WAL service can change the
domain internals to `(owner, export_name)` when auth/client identity exists,
without changing `WalDurableEngine`.

`WalRequest` is deliberately small and should be constructed through a
validator so illegal length combinations are rejected before append:

```rust
pub struct WalRequest {
    pub range: ByteRange,
    pub data: Vec<u8>,
}

impl WalRequest {
    pub fn new(range: ByteRange, data: Vec<u8>) -> Result<Self>;
}
```

`WalRequest::new` requires `data` to be non-empty and to match `range.len`.
Zero-length writes are handled as no-ops by `WalDurableEngine`.

`WalRecord` is the durable append result and the read-view apply input:

```rust
pub struct WalRecord {
    pub seq: WalSeq,
    pub range: ByteRange,
    pub data: Vec<u8>,
}
```

`WalDurableEngine` needs to move write bytes into `WalRequest` without
dropping the admission permit early. Add a small owned admitted-request
boundary to the existing export API. The exact type can be adjusted during
implementation, but the contract should be:

```rust
impl AdmittedExportRequest {
    pub fn into_owned(self) -> OwnedAdmittedExportRequest;
}

pub struct OwnedAdmittedExportRequest {
    request: ExportRequest,
    // Owns the admission permit and context needed for release logging.
}
```

The owned form preserves the same permit lifetime as `AdmittedExportRequest`.
It exists so engines that must take ownership of request bytes can do so
without cloning payloads or bypassing admission-release semantics.

Bounds expose only the WAL state needed by callers:

```rust
pub struct WalBounds {
    pub pruned_through: WalSeq,
    pub last_durable: WalSeq,
}
```

Meanings:

- `pruned_through`: highest sequence known to be unavailable from this WAL
  because local pruning removed its segment prefix;
- `last_durable`: highest sequence durably assigned, including pruned records;
- the next append sequence is derived as `last_durable + 1`.

For a brand-new WAL, `pruned_through = 0` and `last_durable = 0`.

Replay has two explicit forms:

- `replay_after(after)` yields `after < seq <= last_durable` for startup
  recovery;
- `replay_range(after, through)` yields `after < seq <= through` for
  compaction after it captures high watermark `through`.

```rust
pub struct WalReplay {
    // Private implementation detail.
}

impl WalReplay {
    pub async fn next_record(&mut self) -> Result<Option<WalRecord>>;
}
```

Replay fails with a clear WAL-pruned error when `after` is below
`bounds.pruned_through`, because the requested recovery point requires records
that are no longer retained. `replay_range` also fails when `through` is
greater than the durable high watermark or when `through < after`.

Pruning reports the actual cleanup performed:

```rust
pub struct WalPruneResult {
    pub requested_through: WalSeq,
    pub pruned_through: WalSeq,
    pub removed_segments: u64,
}
```

`pruned_through` may be lower than `requested_through` when retained segment
boundaries prevent exact prefix deletion.

## Read View Structures

The first `ExportReadView` should be correctness-first, not an optimized range
cache. It can hold the catalog root/checkpoint plus an in-memory WAL overlay:

```rust
pub struct ExportReadView {
    backing: BackingReaderHandle,
    state: RwLock<ReadViewState>,
}

struct ReadViewState {
    base: RootSnapshot,
    overlay: WalOverlay,
}

pub struct RootSnapshot {
    pub root_node_id: Option<NodeId>,
    pub checkpoint_wal_seq: WalSeq,
    pub size_bytes: u64,
}

struct WalOverlay {
    records_by_seq: BTreeMap<WalSeq, WalRecord>,
}
```

The first overlay implementation may scan `records_by_seq` from newest to
oldest when serving a read, copying bytes from overlapping WAL records and
then filling remaining holes from the backing reader. This is not the final
range index, but it is easy to prove and keeps all required WAL overlay state
in one owner.

The backing reader stays behind a trait:

```rust
pub type BackingReaderHandle = Arc<dyn BackingReader>;

#[async_trait::async_trait]
pub trait BackingReader: Send + Sync {
    async fn read_committed(
        &self,
        root: RootSnapshot,
        range: ByteRange,
    ) -> Result<Vec<u8>>;
}
```

For the first WAL-backed engine, `root_node_id = None` means the backing
reader can be a zero-fill implementation. When COW compaction lands, the same
read-view structure can use a `CommittedTreeReader` without changing
`WalDurableEngine`.

Read-view API:

```rust
impl ExportReadView {
    pub fn new(root: RootSnapshot, backing: BackingReaderHandle) -> Self;

    pub async fn read(&self, range: ByteRange) -> Result<Vec<u8>>;

    pub async fn apply_wal_record(&self, record: WalRecord) -> Result<()>;
}
```

`apply_wal_record` validates the record range against `base.size_bytes` and
rejects records with `seq <= base.checkpoint_wal_seq`.

## Local Segment Format

The local WAL stores one logical export log as ordered segment files under one
export-id directory:

```text
<wal_dir>/<export_id>/<first_seq_padded_16>.wal
```

`export_id` path components must be encoded so arbitrary logical ids cannot
escape `wal_dir`.

Each segment starts with a fixed header:

```text
segment_magic
format_version
header_len
first_seq
header_crc32
```

Each record frame is fixed-header plus inline payload:

```text
record_magic
format_version
header_len
record_kind          -- write in v1
seq
offset
data_len
record_crc32        -- over the record header fields and payload
payload[data_len]
```

The first implementation should use CRC-32C through one focused checksum
dependency. The checksum is for corruption and torn-record detection, not
security.

Payloads are inline. This is acceptable because the protocol already caps one
write payload at `MAX_IO_BYTES`. External payload references are deferred until
there is evidence that inline records are the wrong boundary.

## Local Append

`LocalExportWal` owns an async mutex for the append path.

Append:

```text
lock append mutex
  -> assign seq = bounds.last_durable + 1
  -> rotate segment first if current segment is over target size
  -> sync WAL directory if a new segment was created
  -> encode record frame
  -> write frame and payload
  -> sync segment file
  -> update in-memory bounds
  -> unlock
  -> return WalRecord
```

The initial segment target should be a simple constant, such as 128 MiB. A
single large record may exceed the target; rotation is a boundary for the next
append, not a reason to split a logical write record.

If the process crashes after bytes reach storage but before the NBD reply is
written, replay may apply the record after restart. That is acceptable:
clients that lose the connection before receiving a reply must treat request
completion as unknown.

## Local Recovery And Tail Repair

Opening a local WAL scans segment files in filename order and validates:

- segment headers;
- monotonic and contiguous record sequence numbers;
- record magic and format version;
- payload length bounds;
- record checksums.

A corrupt or partial record is tolerated only at the tail of the newest
segment. The local backend truncates that newest segment back to the last valid
record boundary, syncs the file, and derives the next append position from the
last durable record.

Any corruption or sequence gap before the newest segment tail is a hard WAL
corruption error. The implementation must not silently skip interior records.

## Local Pruning

`LocalExportWal.prune_through(seq)` holds the append mutex. It assumes the
caller has already proven checkpoint coverage and retention-policy safety; the
local backend only enforces WAL-local bounds.

Prune:

```text
validate seq <= bounds.last_durable
  -> if active segment contains only records <= seq:
       create new header-only segment at last_durable + 1
       sync new segment and WAL directory
  -> delete non-active segment files with max_seq <= seq
  -> sync WAL directory best effort
  -> rebuild or update in-memory segment index
```

A header-only active segment is allowed. It preserves the next append position
after all older records have been pruned.

# Ownership And Source Of Truth

Durable truth for a WAL-backed export is:

```text
catalog head at checkpoint C
  + durable WAL records with seq > C
```

Serving truth is:

```text
ExportReadView base root/checkpoint
  + required WAL overlay entries newer than that checkpoint
```

`ExportReadView` is required serving state, but it is not durable truth. It is
rebuilt from catalog plus WAL on open.

`LocalExportWal` owns:

- local WAL path layout;
- segment headers;
- record framing;
- checksums;
- sequence assignment for the local backend;
- local tail repair;
- local segment deletion.

`WalDurableEngine` owns:

- request-path read/write/flush behavior;
- applying durable WAL records to `ExportReadView`;
- holding admission permits through WAL append and read-view apply.

`WalDurableEngine` does not own:

- blob writes;
- tree metadata mutation;
- catalog checkpoint publication;
- compaction output or physical WAL cleanup policy;
- lifecycle compaction source state.

`ExportCatalog` owns:

- export lifecycle;
- current export head;
- committed root/checkpoint;
- future checkpoint publication.

`CompactionManager` owns:

- accepting idempotent compaction jobs;
- owning the separate bounded compaction queue used for background jobs;
- deciding which durable WAL prefix is represented by committed tree state;
- rereading the current catalog head when each job runs;
- writing immutable compaction output;
- publishing the new checkpoint before WAL cleanup is requested;
- treating stale publication attempts as no-ops or retryable replans;
- avoiding duplicate work when an equivalent compaction is already running.

The local active export registry owns active serving state. A lifecycle helper
owns lifecycle events such as close-triggered compaction: quiesce the export,
capture the durable WAL high watermark, enqueue a `CompactionJob`, and then
finish close without waiting for compaction to complete. If background
compaction fails, the WAL remains the durable recovery source for the next
open.

The compaction path should be runnable without a live `WalDurableEngine`.
It should reopen or consult the export WAL through `WalProvider`, read catalog
state through `ExportCatalog`, and write committed COW output through the tree
and blob metadata boundaries.

# Future Remote WAL Service

The remote service should replace `WalProvider` and `ExportWal`, not the NBD
engine contract or registry lifecycle contract.

Future remote behavior:

- the service owns sequence assignment and durable replication;
- `append` returns only after the service's durability policy is satisfied;
- `bounds` reads service-owned high-watermark and retention state;
- `replay` returns a paged or streaming cursor;
- `prune_through` becomes a retention request gated by the same checkpoint and
  active-read-view safety rules;
- local segment file names and checksums disappear from the serving process.

This works because callers depend only on:

```text
append -> durable ordered record
bounds -> explicit retained/durable sequence state
replay -> ordered one-record-at-a-time cursor
prune -> checkpoint and read-view gated retention cleanup
```

# Invariants

- WAL sequence is scoped to one `WalDomain`.
- `WalSeq(0)` is the empty checkpoint before any writes.
- `append` assigns exactly one sequence number for each successful write
  request.
- `append` does not return success until the WAL record is durable under the
  backend's policy.
- Zero-length writes do not append WAL records or consume WAL sequence numbers.
- A write reply is sent only after WAL durability and read-view apply both
  succeed.
- WAL-backed writes do not write blobs, mutate tree metadata, or publish
  catalog checkpoints.
- Replay yields records in strictly increasing sequence order.
- Replay never skips an interior corrupt record.
- A final partial or corrupt tail may be discarded during local recovery.
- `bounds.pruned_through <= bounds.last_durable`.
- `replay_after(after)` and `replay_range(after, through)` are invalid when
  `after < bounds.pruned_through`.
- `prune_through(seq)` must never delete a record with sequence greater than
  `seq`.
- `prune_through(seq)` is safe only after durable committed state represents
  every WAL record with sequence `<= seq` and the retention policy says active
  read views have had enough time to refresh past that prefix.
- Applying a WAL record to `ExportReadView` is derived-state reconstruction,
  not a durability operation.
- `WalDurableEngine` serializes write append-and-apply in v1, so read-view
  apply order matches WAL sequence order.
- `ExportReadView` is the only owner of in-memory WAL overlay state for an
  active WAL-backed export.
- Flush does not require compaction.
- Clean close should trigger compaction through the durable WAL high watermark,
  but failed close-triggered compaction must leave the export recoverable by
  WAL replay.
- Clones or snapshots use committed COW roots and start a new export WAL; they
  do not depend on the source export's current WAL.
- Local segment files are backend detail and must not leak into engine,
  connection, admission, read-view, or catalog APIs.

# Alternatives Considered

## One file per export with in-place prefix rewriting

This is simpler for append, but prefix pruning would require rewriting or
copying the active file. That creates avoidable crash-consistency hazards.
Segment files keep pruning as file deletion.

## One file per export with no pruning

This is the absolute dumbest WAL, but the user explicitly wants
`prune_through`. It also makes replay cost grow without a cleanup boundary.

## Random `get_record(WalSeq)`

Random lookup requires an index or repeated scans. Startup and compaction need
ordered replay, so the first SDK should expose a cursor first. A later service
can add indexed lookup if a real caller appears.

## Flush records

Flush records add recovery surface without improving the first contract. The
admission barrier already makes v1 flush wait behind prior writes that have
completed WAL append and read-view apply.

## Reusing `SimpleDurableEngine`

`SimpleDurableEngine` is intentionally direct-commit and mutable. WAL-backed
serving should be a separate engine path so it can use `ExportReadView`,
future COW checkpoints, and service-backed WAL semantics without weakening the
simple durable baseline.

# Migration / Rollout

Rollout is additive and split along the same three-part boundary.

Part 1 rollout:

- add required `runtime.wal_dir` config, with generated defaults pointing at
  `~/.cache/nbd/wal`;
- add WAL domain structs and the local backend under `nbd-server`;
- keep the WAL module unused by export serving until Part 2.

The first prototype does not need config-file backward compatibility. Existing
hand-written configs may need to add `runtime.wal_dir` after this change.

Part 2 rollout:

- add a `wal_durable` engine kind when the WAL-backed export is ready to serve
  through the registry;
- add `ExportEngineKind::WalDurable`;
- add `ExportLayoutKind::CowImmutableTree`;
- update Prisma schema constraints and add a migration for the new engine and
  layout values;
- update SQLite `create_export` so `wal_durable` creates a
  `cow_immutable_tree` head with `root_node_id = null` and
  `checkpoint_wal_seq = 0`;
- update parsing, display, CLI, catalog, and registry tests for
  `wal_durable`;
- expose `nbdcli create --engine wal_durable`;
- wire `ExportFactory` to open `ExportWalHandle` values for that engine
  kind while keeping `LocalExportRegistry` focused on active lifecycle;
- leave existing `memory` and `simple_durable` exports unchanged;
- do not migrate existing `simple_durable` exports into WAL-backed exports.

The WAL-backed engine should create exports with a COW-compatible head shape:

```text
layout_kind = cow_immutable_tree
root_node_id = null
checkpoint_wal_seq = 0
```

The first WAL series may need to add that layout kind even before compaction
exists. The committed root is initially all zeroes, and WAL replay supplies all
post-checkpoint writes.

Part 3 rollout:

- add a `CompactionManager` boundary that can compact a `wal_durable` export
  from durable catalog, WAL, tree, and blob sources;
- enqueue an idempotent background `CompactionJob` on clean close for the
  first implementation, then let close finish without waiting;
- publish the compacted root/checkpoint through a catalog transaction that
  compares the current `export_heads` row with the head the compaction plan
  used;
- keep close correct when compaction fails by retaining WAL recovery;
- keep WAL cleanup compatible with future time-based retention and periodic
  read-view refresh; Series 3 may defer physical pruning unless a test-owned
  lifecycle can prove pruning is safe.

# Validation Strategy

Part 1 validation proves the standalone WAL primitive:

- record codec tests for round trip, checksum mismatch, bad magic, bad format,
  oversized length, and partial tail;
- local WAL append/reopen tests proving `bounds` and sequence assignment;
- replay tests proving records are yielded one at a time and in order;
- recovery tests proving final partial tail truncation and interior corruption
  failure;
- pruning tests proving only fully covered segments are deleted, active segment
  rotation preserves the next append position, and replay below
  `pruned_through` fails.

Part 1 commands:

```text
cargo test -p nbd-config
cargo test -p nbd-server --test wal
cargo fmt --all --check
```

Part 2 validation proves serving integration:

- engine tests proving write replies require WAL append and read-view apply;
- read-view tests proving newer WAL records override older overlapping bytes
  and uncovered bytes fall back to the backing reader;
- protocol restart tests proving WAL-backed writes survive server restart;
- flush tests proving pipelined writes before flush are visible after flush;
- config tests proving default and explicit `wal_dir` loading;
- catalog and CLI tests proving `wal_durable` creates a `cow_immutable_tree`
  head and leaves `memory` and `simple_durable` behavior unchanged;
- registry tests proving only `wal_durable` opens the WAL-backed engine and
  `Opening` blocks concurrent opens during replay.

Part 2 commands:

```text
cargo test -p nbd-server --test local_export_registry
make test-protocol
cargo test --workspace
cargo fmt --all --check
cargo clippy --workspace --all-targets -- -D warnings
```

Part 3 validation proves lifecycle compaction and cleanup, using close as the
first trigger:

- close-triggered enqueue tests proving clean close submits a compaction job
  through the current durable WAL high watermark and does not wait for that job
  before releasing the export;
- compaction tests proving the worker rereads the current database head before
  planning;
- restart tests proving data is readable after close-triggered compaction
  publishes a checkpoint for the represented WAL prefix;
- failure-injection tests proving close remains recoverable when compaction
  fails before checkpoint publication;
- idempotence tests proving duplicate and stale compaction jobs cannot move the
  checkpoint backward and either no-op or replan;
- cleanup-boundary tests proving `prune_through` is not invoked merely because
  compaction published a checkpoint when retention safety is not established.

Part 3 commands:

```text
cargo test -p nbd-server --test wal
cargo test -p nbd-server --test tcp_integration
make test-protocol
cargo test --workspace
cargo fmt --all --check
cargo clippy --workspace --all-targets -- -D warnings
```

# Risks

- Replay can be memory-heavy in v1 because records may be collected into a
  `Vec` internally. The cursor API keeps this replaceable.
- Local `sync_all` can be slow. Correctness matters more than throughput for
  this primitive.
- Filesystems and disks can lie about durability. The local WAL is not a
  substitute for replicated storage.
- Large WAL segments may delay physical pruning when the prune point falls
  inside a segment.
- If compaction or another durable checkpoint path calls `prune_through` too
  early, recovery can lose data. This must remain a hard API contract.
- If close-triggered compaction fails, replay cost remains until a later
  successful compaction. That is preferable to making close depend on cleanup.
- If physical cleanup ignores the retention and refresh contract, a stale
  serving view can lose the WAL prefix it still needs. Series 3 should preserve
  the future retention boundary rather than inventing an incompatible local
  active-view cleanup policy.
- If compaction plans from stale catalog state can still publish, racing jobs
  can regress the head or expose incomplete committed data. Publication must
  compare against the database head in the same transaction that inserts
  metadata and advances `export_heads`.
- A write whose reply is lost during crash may still replay after restart.
  This is acceptable but should be documented as unknown completion after
  disconnect.

# Design Exit Criteria

This design is ready for `$review-plan` when:

- the `WalDomain`-scoped `WalProvider` / `ExportWal` SDK shape is accepted;
- the three-part build boundary is accepted: standalone WAL first, serving
  integration second, lifecycle compaction and cleanup third;
- the `LocalExportRegistry` / `WalDurableEngine` / `ExportReadView`
  integration structure is accepted;
- `bounds`, `replay`, and `prune_through` semantics are accepted;
- local segment files are accepted as the simplest correct pruning mechanism;
- the tail-repair policy is accepted;
- `WalDurableEngine` is accepted as separate from `SimpleDurableEngine`;
- compaction is accepted as a lifecycle event, with close as the first trigger
  and normal cleanup path;
- the memory-backed first `WalReplay` implementation is accepted as a v1
  limitation behind a cursor API.
